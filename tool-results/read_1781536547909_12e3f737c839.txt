    76ŌåÆ    76ŌåÆ- `heading-hierarchy` - Sequential h1ŌåÆh6, no level skip

    85ŌåÆ    85ŌåÆ

    95ŌåÆ    95ŌåÆ- `system-gestures` - Don't block system gestures (Control Center, back swipe, etc.) (Apple HIG)
    96ŌåÆ    96ŌåÆ- `press-feedback` - Visual feedback on press (ripple/highlight; MD state layers)

   180ŌåÆ   180ŌåÆ
   181ŌåÆ   181ŌåÆ- `duration-timing` - Use 150ŌĆō300ms for micro-interactions; complex transitions Ōēż400ms; avoid >500ms (MD)
   182ŌåÆ   182ŌåÆ- `transform-performance` - Use transform/opacity only; avoid animating width/height/top/left

   187ŌåÆ   187ŌåÆ- `state-transition` - State changes (hover / active / expanded / collapsed / modal) should animate smoothly, not snap

   189ŌåÆ   189ŌåÆ- `parallax-subtle` - Use parallax sparingly; must respect reduced-motion and not cause disorientation (Apple HIG)
   190ŌåÆ   190ŌåÆ- `spring-physics` - Prefer spring/physics-based curves over linear or cubic-bezier for natural feel (Apple HIG fluid animations)
   191ŌåÆ   191ŌåÆ- `exit-faster-than-enter` - Exit animations shorter than enter (~60ŌĆō70% of enter duration) to feel responsive (MD motion)
   192ŌåÆ   192ŌåÆ- `stagger-sequence` - Stagger list/grid item entrance by 30ŌĆō50ms per item; avoid all-at-once or too-slow reveals (MD)
   193ŌåÆ   193ŌåÆ- `shared-element-transition` - Use shared element / hero transitions for visual continuity between screens (MD, HIG)
   194ŌåÆ   194ŌåÆ- `interruptible` - Animations must be interruptible; user tap/gesture cancels in-progress animation immediately (Apple HIG)
   195ŌåÆ   195ŌåÆ- `no-blocking-animation` - Never block user input during an animation; UI must stay interactive (Apple HIG)
   196ŌåÆ   196ŌåÆ- `fade-crossfade` - Use crossfade for content replacement within the same container (MD)
   197ŌåÆ   197ŌåÆ- `scale-feedback` - Subtle scale (0.95ŌĆō1.05) on press for tappable cards/buttons; restore on release (HIG, MD)
   198ŌåÆ   198ŌåÆ- `gesture-feedback` - Drag, swipe, and pinch must provide real-time visual response tracking the finger (MD Motion)
   199ŌåÆ   199ŌåÆ- `hierarchy-motion` - Use translate/scale direction to express hierarchy: enter from below = deeper, exit upward = back (MD)
   200ŌåÆ   200ŌåÆ- `motion-consistency` - Unify duration/easing tokens globally; all animations share the same rhythm and feel

   202ŌåÆ   202ŌåÆ- `modal-motion` - Modals/sheets should animate from their trigger source (scale+fade or slide-in) for spatial context (HIG, MD)
   203ŌåÆ   203ŌåÆ- `navigation-direction` - Forward navigation animates left/up; backward animates right/down ŌĆö keep direction logically consistent (HIG)
   204ŌåÆ   204ŌåÆ- `layout-shift-avoid` - Animations must not cause layout reflow or CLS; use transform for position changes

   221ŌåÆ   221ŌåÆ- `autofill-support` - Use autocomplete / textContentType attributes so the system can autofill (HIG, MD)
   222ŌåÆ   222ŌåÆ- `undo-support` - Allow undo for destructive or bulk actions (e.g. "Undo delete" toast) (Apple HIG)
   223ŌåÆ   223ŌåÆ- `success-feedback` - Confirm completed actions with brief visual feedback (checkmark, toast, color flash) (MD)
   224ŌåÆ   224ŌåÆ- `error-recovery` - Error messages must include a clear recovery path (retry, edit, help link) (HIG, MD)
   225ŌåÆ   225ŌåÆ- `multi-step-progress` - Multi-step flows show step indicator or progress bar; allow back navigation (MD)
   226ŌåÆ   226ŌåÆ- `form-autosave` - Long forms should auto-save drafts to prevent data loss on accidental dismissal (Apple HIG)
   227ŌåÆ   227ŌåÆ- `sheet-dismiss-confirm` - Confirm before dismissing a sheet/modal with unsaved changes (Apple HIG)
   228ŌåÆ   228ŌåÆ- `error-clarity` - Error messages must state cause + how to fix (not just "Invalid input") (HIG, MD)

   230ŌåÆ   230ŌåÆ- `read-only-distinction` - Read-only state should be visually and semantically different from disabled (MD)
   231ŌåÆ   231ŌåÆ- `focus-management` - After submit error, auto-focus the first invalid field (WCAG, MD)
   232ŌåÆ   232ŌåÆ- `error-summary` - For multiple errors, show summary at top with anchor links to each field (WCAG)
   233ŌåÆ   233ŌåÆ- `touch-friendly-input` - Mobile input height Ōēź44px to meet touch target requirements (Apple HIG)
   234ŌåÆ   234ŌåÆ- `destructive-emphasis` - Destructive actions use semantic danger color (red) and are visually separated from primary actions (HIG, MD)
   235ŌåÆ   235ŌåÆ- `toast-accessibility` - Toasts must not steal focus; use aria-live="polite" for screen reader announcement (WCAG)
   236ŌåÆ   236ŌåÆ- `aria-live-errors` - Form errors use aria-live region or role="alert" to notify screen readers (WCAG)
   237ŌåÆ   237ŌåÆ- `contrast-feedback` - Error and success state colors must meet 4.5:1 contrast ratio (WCAG, MD)
   238ŌåÆ   238ŌåÆ- `timeout-feedback` - Request timeout must show clear feedback with retry option (MD)

   240ŌåÆ   240ŌåÆ### 9. Navigation Patterns (HIGH)
   241ŌåÆ   241ŌåÆ
   242ŌåÆ   242ŌåÆ- `bottom-nav-limit` - Bottom navigation max 5 items; use labels with icons (Material Design)
   243ŌåÆ   243ŌåÆ- `drawer-usage` - Use drawer/sidebar for secondary navigation, not primary actions (Material Design)
   244ŌåÆ   244ŌåÆ- `back-behavior` - Back navigation must be predictable and consistent; preserve scroll/state (Apple HIG, MD)
   245ŌåÆ   245ŌåÆ- `deep-linking` - All key screens must be reachable via deep link / URL for sharing and notifications (Apple HIG, MD)
   246ŌåÆ   246ŌåÆ- `tab-bar-ios` - iOS: use bottom Tab Bar for top-level navigation (Apple HIG)
   247ŌåÆ   247ŌåÆ- `top-app-bar-android` - Android: use Top App Bar with navigation icon for primary structure (Material Design)
   248ŌåÆ   248ŌåÆ- `nav-label-icon` - Navigation items must have both icon and text label; icon-only nav harms discoverability (MD)
   249ŌåÆ   249ŌåÆ- `nav-state-active` - Current location must be visually highlighted (color, weight, indicator) in navigation (HIG, MD)
   250ŌåÆ   250ŌåÆ- `nav-hierarchy` - Primary nav (tabs/bottom bar) vs secondary nav (drawer/settings) must be clearly separated (MD)
   251ŌåÆ   251ŌåÆ- `modal-escape` - Modals and sheets must offer a clear close/dismiss affordance; swipe-down to dismiss on mobile (Apple HIG)
   252ŌåÆ   252ŌåÆ- `search-accessible` - Search must be easily reachable (top bar or tab); provide recent/suggested queries (MD)
   253ŌåÆ   253ŌåÆ- `breadcrumb-web` - Web: use breadcrumbs for 3+ level deep hierarchies to aid orientation (MD)
   254ŌåÆ   254ŌåÆ- `state-preservation` - Navigating back must restore previous scroll position, filter state, and input (HIG, MD)
   255ŌåÆ   255ŌåÆ- `gesture-nav-support` - Support system gesture navigation (iOS swipe-back, Android predictive back) without conflict (HIG, MD)
   256ŌåÆ   256ŌåÆ- `tab-badge` - Use badges on nav items sparingly to indicate unread/pending; clear after user visits (HIG, MD)
   257ŌåÆ   257ŌåÆ- `overflow-menu` - When actions exceed available space, use overflow/more menu instead of cramming (MD)
   258ŌåÆ   258ŌåÆ- `bottom-nav-top-level` - Bottom nav is for top-level screens only; never nest sub-navigation inside it (MD)
   259ŌåÆ   259ŌåÆ- `adaptive-navigation` - Large screens (Ōēź1024px) prefer sidebar; small screens use bottom/top nav (Material Adaptive)
   260ŌåÆ   260ŌåÆ- `back-stack-integrity` - Never silently reset the navigation stack or unexpectedly jump to home (HIG, MD)
   261ŌåÆ   261ŌåÆ- `navigation-consistency` - Navigation placement must stay the same across all pages; don't change by page type
   262ŌåÆ   262ŌåÆ- `avoid-mixed-patterns` - Don't mix Tab + Sidebar + Bottom Nav at the same hierarchy level
   263ŌåÆ   263ŌåÆ- `modal-vs-navigation` - Modals must not be used for primary navigation flows; they break the user's path (HIG)
   264ŌåÆ   264ŌåÆ- `focus-on-route-change` - After page transition, move focus to main content region for screen reader users (WCAG)
   265ŌåÆ   265ŌåÆ- `persistent-nav` - Core navigation must remain reachable from deep pages; don't hide it entirely in sub-flows (HIG, MD)
   266ŌåÆ   266ŌåÆ- `destructive-nav-separation` - Dangerous actions (delete account, logout) must be visually and spatially separated from normal nav items (HIG, MD)
   267ŌåÆ   267ŌåÆ- `empty-nav-state` - When a nav destination is unavailable, explain why instead of silently hiding it (MD)

   269ŌåÆ   269ŌåÆ### 10. Charts & Data (LOW)
   270ŌåÆ   270ŌåÆ
   271ŌåÆ   271ŌåÆ- `chart-type` - Match chart type to data type (trend ŌåÆ line, comparison ŌåÆ bar, proportion ŌåÆ pie/donut)
   272ŌåÆ   272ŌåÆ- `color-guidance` - Use accessible color palettes; avoid red/green only pairs for colorblind users (WCAG, MD)

   276ŌåÆ   276ŌåÆ- `tooltip-on-interact` - Provide tooltips/data labels on hover (Web) or tap (mobile) showing exact values (HIG, MD)

   278ŌåÆ   278ŌåÆ- `responsive-chart` - Charts must reflow or simplify on small screens (e.g. horizontal bar instead of vertical, fewer ticks)
   279ŌåÆ   279ŌåÆ- `empty-data-state` - Show meaningful empty state when no data exists ("No data yet" + guidance), not a blank chart (MD)

   283ŌåÆ   283ŌåÆ- `number-formatting` - Use locale-aware formatting for numbers, dates, currencies on axes and labels (HIG, MD)
   284ŌåÆ   284ŌåÆ- `touch-target-chart` - Interactive chart elements (points, segments) must have Ōēź44pt tap area or expand on touch (Apple HIG)
   285ŌåÆ   285ŌåÆ- `no-pie-overuse` - Avoid pie/donut for >5 categories; switch to bar chart for clarity
   286ŌåÆ   286ŌåÆ- `contrast-data` - Data lines/bars vs background Ōēź3:1; data text labels Ōēź4.5:1 (WCAG)
   287ŌåÆ   287ŌåÆ- `legend-interactive` - Legends should be clickable to toggle series visibility (MD)
   288ŌåÆ   288ŌåÆ- `direct-labeling` - For small datasets, label values directly on the chart to reduce eye travel
   289ŌåÆ   289ŌåÆ- `tooltip-keyboard` - Tooltip content must be keyboard-reachable and not rely on hover alone (WCAG)
   290ŌåÆ   290ŌåÆ- `sortable-table` - Data tables must support sorting with aria-sort indicating current sort state (WCAG)
   291ŌåÆ   291ŌåÆ- `axis-readability` - Axis ticks must not be cramped; maintain readable spacing, auto-skip on small screens
   292ŌåÆ   292ŌåÆ- `data-density` - Limit information density per chart to avoid cognitive overload; split into multiple charts if needed
   293ŌåÆ   293ŌåÆ- `trend-emphasis` - Emphasize data trends over decoration; avoid heavy gradients/shadows that obscure the data
   294ŌåÆ   294ŌåÆ- `gridline-subtle` - Grid lines should be low-contrast (e.g. gray-200) so they don't compete with data
   295ŌåÆ   295ŌåÆ- `focusable-elements` - Interactive chart elements (points, bars, slices) must be keyboard-navigable (WCAG)
   296ŌåÆ   296ŌåÆ- `screen-reader-summary` - Provide a text summary or aria-label describing the chart's key insight for screen readers (WCAG)
   297ŌåÆ   297ŌåÆ- `error-state-chart` - Data load failure must show error message with retry action, not a broken/empty chart
   298ŌåÆ   298ŌåÆ- `export-option` - For data-heavy products, offer CSV/image export of chart data
   299ŌåÆ   299ŌåÆ- `drill-down-consistency` - Drill-down interactions must maintain a clear back-path and hierarchy breadcrumb
   300ŌåÆ   300ŌåÆ- `time-scale-clarity` - Time series charts must clearly label time granularity (day/week/month) and allow switching

   559ŌåÆ   559ŌåÆ## Common Rules for Professional UI
   560ŌåÆ   560ŌåÆ
   561ŌåÆ   561ŌåÆThese are frequently overlooked issues that make UI look unprofessional:
   562ŌåÆ   562ŌåÆScope notice: The rules below are for App UI (iOS/Android/React Native/Flutter), not desktop-web interaction patterns.
   563ŌåÆ   563ŌåÆ
   564ŌåÆ   564ŌåÆ### Icons & Visual Elements
   565ŌåÆ   565ŌåÆ
   566ŌåÆ   566ŌåÆ| Rule | Standard | Avoid | Why It Matters |
   567ŌåÆ   567ŌåÆ|------|----------|--------|----------------|
   568ŌåÆ   568ŌåÆ| **No Emoji as Structural Icons** | Use vector-based icons (e.g., Lucide, react-native-vector-icons, @expo/vector-icons). | Using emojis (­¤Ä© ­¤ÜĆ ŌÜÖ’ĖÅ) for navigation, settings, or system controls. | Emojis are font-dependent, inconsistent across platforms, and cannot be controlled via design tokens. |
   569ŌåÆ   569ŌåÆ| **Vector-Only Assets** | Use SVG or platform vector icons that scale cleanly and support theming. | Raster PNG icons that blur or pixelate. | Ensures scalability, crisp rendering, and dark/light mode adaptability. |
   570ŌåÆ   570ŌåÆ| **Stable Interaction States** | Use color, opacity, or elevation transitions for press states without changing layout bounds. | Layout-shifting transforms that move surrounding content or trigger visual jitter. | Prevents unstable interactions and preserves smooth motion/perceived quality on mobile. |
   571ŌåÆ   571ŌåÆ| **Correct Brand Logos** | Use official brand assets and follow their usage guidelines (spacing, color, clear space). | Guessing logo paths, recoloring unofficially, or modifying proportions. | Prevents brand misuse and ensures legal/platform compliance. |
   572ŌåÆ   572ŌåÆ| **Consistent Icon Sizing** | Define icon sizes as design tokens (e.g., icon-sm, icon-md = 24pt, icon-lg). | Mixing arbitrary values like 20pt / 24pt / 28pt randomly. | Maintains rhythm and visual hierarchy across the interface. |
   573ŌåÆ   573ŌåÆ| **Stroke Consistency** | Use a consistent stroke width within the same visual layer (e.g., 1.5px or 2px). | Mixing thick and thin stroke styles arbitrarily. | Inconsistent strokes reduce perceived polish and cohesion. |
   574ŌåÆ   574ŌåÆ| **Filled vs Outline Discipline** | Use one icon style per hierarchy level. | Mixing filled and outline icons at the same hierarchy level. | Maintains semantic clarity and stylistic coherence. |
   575ŌåÆ   575ŌåÆ| **Touch Target Minimum** | Minimum 44├Ś44pt interactive area (use hitSlop if icon is smaller). | Small icons without expanded tap area. | Meets accessibility and platform usability standards. |
   576ŌåÆ   576ŌåÆ| **Icon Alignment** | Align icons to text baseline and maintain consistent padding. | Misaligned icons or inconsistent spacing around them. | Prevents subtle visual imbalance that reduces perceived quality. |
   577ŌåÆ   577ŌåÆ| **Icon Contrast** | Follow WCAG contrast standards: 4.5:1 for small elements, 3:1 minimum for larger UI glyphs. | Low-contrast icons that blend into the background. | Ensures accessibility in both light and dark modes. |
   578ŌåÆ   578ŌåÆ
   579ŌåÆ   579ŌåÆ
   580ŌåÆ   580ŌåÆ### Interaction (App)
   581ŌåÆ   581ŌåÆ
   582ŌåÆ   582ŌåÆ| Rule | Do | Don't |
   583ŌåÆ   583ŌåÆ|------|----|----- |
   584ŌåÆ   584ŌåÆ| **Tap feedback** | Provide clear pressed feedback (ripple/opacity/elevation) within 80-150ms | No visual response on tap |
   585ŌåÆ   585ŌåÆ| **Animation timing** | Keep micro-interactions around 150-300ms with platform-native easing | Instant transitions or slow animations (>500ms) |
   586ŌåÆ   586ŌåÆ| **Accessibility focus** | Ensure screen reader focus order matches visual order and labels are descriptive | Unlabeled controls or confusing focus traversal |
   587ŌåÆ   587ŌåÆ| **Disabled state clarity** | Use disabled semantics (`disabled`/native disabled props), reduced emphasis, and no tap action | Controls that look tappable but do nothing |
   588ŌåÆ   588ŌåÆ| **Touch target minimum** | Keep tap areas >=44x44pt (iOS) or >=48x48dp (Android), expand hit area when icon is smaller | Tiny tap targets or icon-only hit areas without padding |
   589ŌåÆ   589ŌåÆ| **Gesture conflict prevention** | Keep one primary gesture per region and avoid nested tap/drag conflicts | Overlapping gestures causing accidental actions |
   590ŌåÆ   590ŌåÆ| **Semantic native controls** | Prefer native interactive primitives (`Button`, `Pressable`, platform equivalents) with proper accessibility roles | Generic containers used as primary controls without semantics |
   591ŌåÆ   591ŌåÆ
   592ŌåÆ   592ŌåÆ### Light/Dark Mode Contrast
   593ŌåÆ   593ŌåÆ
   594ŌåÆ   594ŌåÆ| Rule | Do | Don't |
   595ŌåÆ   595ŌåÆ|------|----|----- |
   596ŌåÆ   596ŌåÆ| **Surface readability (light)** | Keep cards/surfaces clearly separated from background with sufficient opacity/elevation | Overly transparent surfaces that blur hierarchy |
   597ŌåÆ   597ŌåÆ| **Text contrast (light)** | Maintain body text contrast >=4.5:1 against light surfaces | Low-contrast gray body text |
   598ŌåÆ   598ŌåÆ| **Text contrast (dark)** | Maintain primary text contrast >=4.5:1 and secondary text >=3:1 on dark surfaces | Dark mode text that blends into background |
   599ŌåÆ   599ŌåÆ| **Border and divider visibility** | Ensure separators are visible in both themes (not just light mode) | Theme-specific borders disappearing in one mode |
   600ŌåÆ   600ŌåÆ| **State contrast parity** | Keep pressed/focused/disabled states equally distinguishable in light and dark themes | Defining interaction states for one theme only |
   601ŌåÆ   601ŌåÆ| **Token-driven theming** | Use semantic color tokens mapped per theme across app surfaces/text/icons | Hardcoded per-screen hex values |
   602ŌåÆ   602ŌåÆ| **Scrim and modal legibility** | Use a modal scrim strong enough to isolate foreground content (typically 40-60% black) | Weak scrim that leaves background visually competing |
   603ŌåÆ   603ŌåÆ
   604ŌåÆ   604ŌåÆ### Layout & Spacing
   605ŌåÆ   605ŌåÆ
   606ŌåÆ   606ŌåÆ| Rule | Do | Don't |
   607ŌåÆ   607ŌåÆ|------|----|----- |
   608ŌåÆ   608ŌåÆ| **Safe-area compliance** | Respect top/bottom safe areas for all fixed headers, tab bars, and CTA bars | Placing fixed UI under notch, status bar, or gesture area |
   609ŌåÆ   609ŌåÆ| **System bar clearance** | Add spacing for status/navigation bars and gesture home indicator | Let tappable content collide with OS chrome |
   610ŌåÆ   610ŌåÆ| **Consistent content width** | Keep predictable content width per device class (phone/tablet) | Mixing arbitrary widths between screens |
   611ŌåÆ   611ŌåÆ| **8dp spacing rhythm** | Use a consistent 4/8dp spacing system for padding/gaps/section spacing | Random spacing increments with no rhythm |
   612ŌåÆ   612ŌåÆ| **Readable text measure** | Keep long-form text readable on large devices (avoid edge-to-edge paragraphs on tablets) | Full-width long text that hurts readability |
   613ŌåÆ   613ŌåÆ| **Section spacing hierarchy** | Define clear vertical rhythm tiers (e.g., 16/24/32/48) by hierarchy | Similar UI levels with inconsistent spacing |
   614ŌåÆ   614ŌåÆ| **Adaptive gutters by breakpoint** | Increase horizontal insets on larger widths and in landscape | Same narrow gutter on all device sizes/orientations |
   615ŌåÆ   615ŌåÆ| **Scroll and fixed element coexistence** | Add bottom/top content insets so lists are not hidden behind fixed bars | Scroll content obscured by sticky headers/footers |
